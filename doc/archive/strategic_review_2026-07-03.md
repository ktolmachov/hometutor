# Стратегический обзор hometutor — что развивать, а что нет

**Дата:** 2026-07-03 · **Ревизия:** 2026-07-05 (после встречного аудита; метрики пересняты по текущему дереву, формулировки уточнены — журнал ревизии в конце документа).
**Тип:** глубокий анализ приложения, архитектуры, документации, тестов, зависимостей и гигиены репозитория.
**Предыдущий аудит:** [arch_review_2026-06-25.md](arch_review_2026-06-25.md) (часть находок закрыта регрессионными гвардами — учтено ниже).
**Размещение:** `doc/archive/` — сознательно вне runtime-`docs/`, по прецеденту `arch_review_2026-06-25.md`: это датированный процессный срез, а не эксплуатационный документ. Если политика `docs/index.md` («`doc/*` — legacy») будет ужесточаться, оба обзора переезжают вместе.

## TL;DR

Проект в необычно хорошей форме для своего размера: границы архитектуры чистые и **охраняются автоматическими гвардами**, документация системная и актуализируется по датам, тесты зелёные и быстрые (на ревизию — 269 за ~5 сек), workflow CI/CD настроен. Главные риски — не в качестве, а в **масштабе и расползании**: ~86 700 строк кода в 360 модулях на локальное однопользовательское приложение, 213 полей конфигурации, 27 модулей только под Smart Study Router, и гвард размера **уже сломан текущими незакоммиченными правками**.

Стратегический вывод: **прекратить наращивать поверхность (новые фичи, ML, интеграции) и инвестировать в ядро учебного цикла, консолидацию и тесты сервисного слоя.**

## Что уже сильно — продолжать в том же духе

**1. Самоохраняющаяся архитектура.** Самое ценное в проекте. После аудита 25.06 появились регрессионные гварды (`scripts/arch_regression_guards.py`): контроль доступа к конфигу (`check_config_access.py`), детектор мёртвых модулей (`check_dead_modules.py`), проверка requirements против импортов (`check_requirements_imports.py`), бюджет размера (`check_size_budget.py`). Границы подтверждаются: все LLM-клиенты через `app/provider.py`, промпты централизованы в `app/prompts/`, backend не импортирует UI, никакого ad-hoc SQL в роутерах/UI, SQL-идентификаторы через allowlist. **Продолжать: каждое новое архитектурное правило — сразу с гвардом.**

**2. Документационная система.** `docs/index.md` как навигатор с ролями, таблица «источников истины», явная политика документации, датированные актуализации с журналом исправленного дрейфа. Уровень зрелых команд. **Продолжать актуализацию при каждом изменении маршрутов/поведения.**

**3. Ядро продукта — учебный цикл.** Ответ с источниками → tutor → quiz → flashcards → SM-2 → mastery → адаптивный план → SSR. Цельная, отличимая от «чата с PDF» идея. **Всё развитие должно углублять этот цикл, а не добавлять параллельные поверхности.**

**4. Работа над видимостью UI (в процессе).** `app/ui/feature_registry.py` с 4 уровнями опыта, `docs/first_run_experience_plan.md`, Mission Control с preflight-проверками. Правильный ответ на главную продуктовую проблему — перегруз интерфейса ~19 фичами. **Довести до конца в первую очередь.**

**5. Аутентификация и безопасность.** Опциональный JWT + bcrypt с prod-гвардом против dev-секрета (`guard_insecure_jwt_secret`), изоляция state через `contextvars` + резолв пути без переписывания схемы — элегантно. Log-masking подключён и покрыт тестом (`tests/test_logging_invariants.py`). Confirmation-токены на деструктивные скрипты. Local-first с loopback-дефолтами. **Достаточно — не наращивать (роли, OAuth и т.п. этому продукту не нужны).**

**6. CI/CD.** Workflow настроен корректно: ruff + pytest на каждый push, автодеплой на HF Spaces с мягким скипом без секретов. Оговорка: по локальным источникам подтверждена только конфигурация, а не факт успешных прогонов на GitHub (`docs/compliance_upgrade_plan.md` ранее фиксировал «Actions не подтверждён») — проверить страницу Actions/badge и после первого зелёного run считать пункт закрытым.

## Что развивать в первую очередь (приоритетный список)

**П1. Починить сломанный гвард размера — и не «поднятием бюджета».** Прямо сейчас `check_size_budget.py` падает: незакоммиченные правки в `app/ui/mission_control.py` и рост `app/ui/knowledge_graph_d3.py` (1720 строк) превысили бюджеты. Судя по значениям, бюджеты выставлены «по пиковому нарушителю» — храповик крутится не в ту сторону. Нужно: при каждом касании файла-нарушителя выносить куски и **снижать** бюджет после каждого улучшения.

**П2. Тесты сервисного слоя.** 3 500 строк тестов на 86 700 строк кода. Суть не в том, что тесты «в основном UI-smoke» (алгоритмических хватает: scheduling, section index, retrieval budget, guardrails, auth), а в том, что есть **явные дыры вокруг самых тяжёлых сервисов**. Целевых тестов заслуживают:
- `app/query_service.py` (960 строк);
- `app/tutor_orchestrator.py` (696);
- `app/flashcard_service.py` (994, включая SM-2 edge cases);
- `app/index_lifecycle.py` / `app/index_registry.py` — целостность индекса — единственные данные, которые больно терять.

**П3. Консолидация вместо роста.** Конкретные цели:
- **SSR: 27 модулей** (`ssr_*`, `smart_study_*`, `ssr_ai/`) — при задекларированном «deterministic-first». Слить экспериментальные/gated-слои, оставить: сигналы → скоринг → объяснение.
- **Eval: 8 модулей** (`eval_baseline`, `eval_uplift`, `eval_ragas_backend`, `eval_retrieval_comparison`, `router_eval`, `compare_eval`…) — один harness с бэкендами вместо шести параллельных.
- **«Мёртвые с keep-reason» модули**: `check_dead_modules.py` превратился в белый список кладбища — 16 модулей числятся «на будущее» (`ssr_pregeneration`, `answer_parser`, `tutor_context_parser`, «kept for API evolution»…). Провести owner-аудит по одному и удалить неоправданные: git хранит историю, а каждый лишний модуль дорожает при каждом ревью. Исключение — `app/index_backup.py`: он закреплён архитектурным правилом (`docs/conventions_architecture.md`, §index backup/restore), его судьба решается вместе с правилом, не в общей чистке.

**П4. Сжать конфигурационную поверхность.** 213 annotated-полей настроек и 74 env-ключа — поверхность отказов и документационного дрейфа. Пройтись по флагам, которые всегда в дефолте, и убрать вместе с ветками кода. Эвристика: если флаг не переключался ни разу с момента создания — это не конфигурация, а мёртвый код с усложнённым интерфейсом.

**П5. Единый источник зависимостей.** Сейчас два: `requirements.txt` (всё вперемешку, включая optional) и `pyproject.toml` (аккуратные extras: voice/ocr/otel/telegram/ml). CI ставит полный `requirements.txt` — включая optional docling, pymupdf, voice-пакеты — на каждый прогон. Перевести CI и README на `pip install -e .[dev]`; `requirements.txt` либо удалить, либо генерировать. Две поправки к переходу: (а) `ruff` не входит в extra `dev` — добавить его туда (или оставить отдельным шагом установки), иначе lint-шаг сломается; (б) `FlagEmbedding` — это **core**-зависимость в `pyproject`, а не optional: переход на extras сам по себе её из CI не уберёт; если хочется лёгкого CI, нужно отдельное решение по reranker'у (ленивый импорт / extra `rerank`).

## Что точно НЕ нужно развивать (stop-list)

**1. ML-надстройки над SSR.** `ssr_ml_reranking`, `ssr_ml_monitoring`, `scripts/ml/` с обучением forgetting curve — ML-инфраструктура промышленного масштаба для **одного пользователя с крошечным датасетом**. Кривая забывания одного человека не даст данных для обучения ничего лучше SM-2, который уже есть. Рекомендация — именно **заморозить** (не удалять походя): ML-слой подключён как gated enrichment (`_apply_ssr_ml_hybrid_if_enabled` в `smart_study_router.py`, флаг off по умолчанию), это архитектурно допустимый путь, и его демонтаж требует отдельного compatibility-аудита. Просто не вкладывать туда ни строчки, пока детерминированный контур не упрётся в потолок.

**2. Voice-фича.** `app/voice_service.py` + SpeechRecognition/pyttsx3 (extra `voice`). Ценность для учебного RAG-приложения маргинальная, зависимости тяжёлые и платформозависимые. Кандидат на удаление, но это мини-проект, а не один файл: зацепки в `app/ui/sidebar.py`, `app/ui/feature_registry.py`, Telegram voice-handler в `app/telegram_handlers.py`, extras в `pyproject.toml`/`requirements.txt` и упоминания в доках — сносить всё вместе.

**3. Telegram-бот как вторая продуктовая поверхность.** Каждая фича должна работать в двух UI. Если бот не используется ежедневно — заморозить на уровне «уведомления о повторениях», учебный цикл туда не переносить.

**4. Enterprise-обсервабилити.** 8 модулей `metrics_*`, SLO, Langfuse-экспорт, OTel-трейсинг, `latency_budget`, `pipeline_profiler` — оборудование самолёта на велосипеде. Не удалять то, что подключено и работает, но **не расширять**: ни новых SLO-дашбордов, ни новых экспортёров.

**5. Новые режимы retrieval и новые ручки.** Документация сама формулирует правило: «Do not add public retrieval modes when a bounded RAG profile is enough». Соблюдать. Hybrid (Chroma+BM25) + reranker + graph expansion уже выше среднего по качеству.

**6. Новые views в Streamlit UI.** 107 UI-модулей, 19 фич в реестре, и главная текущая работа — *прятать* сложность за уровнями опыта. Добавление 20-й view работает против этой цели. UI-бюджет тратить на консолидацию и полировку tier 1–2.

**7. Расширение auth.** Опциональный JWT сделан для рубрики проектной работы и сделан хорошо. Роли, OAuth, refresh-токены, admin-панели — не нужны local-first продукту.

## Гигиена (исправить один раз и забыть)

1. **48.7 МБ скриншотов в git** (`docs/screenshots/`, 209 файлов, 91.7% tracked-байт репозитория). Перенести в `hometutor-studio` или в Git LFS/release assets; в `docs/quickstart_demo.md` оставить ссылки. Важно: это не чисто механическая чистка — `docs/index.md` и `docs/architecture.md` прямо называют `docs/screenshots/final/` локальной demo-витриной runtime-репозитория, поэтому сначала меняется doc-policy (index/architecture/README/quickstart_demo), потом переносятся файлы.
2. **Сообщения коммитов-числа** («56», «55»). История не помогает ни bisect'у, ни ревью. Писать `тип(область): что и зачем` — примеры в репо уже есть (`docs: sync architecture...`).
3. **Глобальные ruff-игноры маскируют реальные проблемы.** `F821` (undefined name!) и `F401/F403/F405` выключены на весь репозиторий из-за star-реэкспортов в двух файлах (`app/api_models.py`, `app/user_state.py`). Перенести в `per-file-ignores` — сейчас линтер слеп к опечаткам в именах во всех 360 модулях.
4. **Процессные документы в runtime-репо вопреки собственной политике.** `docs/index.md` декларирует: процессные материалы — в `hometutor-studio`. При этом в `docs/` лежат `compliance_upgrade_plan.md`, `first_run_experience_plan.md`, `ui_visibility_phase1_plan.md` — handoff-задания. По завершении работ — в архив или в studio.
5. Мелочи: `.codex_tmp/` в рабочем дереве (не добавлен в `.gitignore`); кириллические имена файлов страниц (`3_Мой_прогресс.py`) дают quoted-пути в git на Windows; `models/` — пустая папка с `.gitkeep`.

## Рекомендуемый порядок действий

1. **Сейчас:** довести до конца работу по видимости UI / first-run (в процессе, незакоммичена) и починить упавший size-гвард в этих же файлах.
2. **Ближайший спринт:** удалить мёртвый кластер (voice, keep-reason-модули), слить eval-harness'ы, перевести CI на `pyproject` extras, вынести скриншоты из git.
3. **Фоново, по правилу «тронул — улучшил»:** дробление файлов >600 строк со снижением бюджета, тесты на `query_service` / `tutor_orchestrator` / `flashcard_service` / `index_lifecycle`.
4. **Постоянно:** любое предложение новой фичи проверять вопросом «углубляет ли это цикл ответ→tutor→quiz→cards→progress→SSR?» Если нет — в studio-backlog, не в код.

---

**Суть в одном предложении:** у проекта отличный скелет и работающая иммунная система (гварды, доки, CI) — угроза не в качестве, а в том, что тело растёт быстрее, чем скелет успевает его держать. Побеждать надо не добавлением, а вычитанием.

## Журнал ревизии

**2026-07-05 — правки по встречному аудиту.** Аудит подтвердил ключевые выводы (падающий size-гвард, 98 маршрутов, 91.7% веса репо в скриншотах) и нашёл устаревшие/неточные места. Исправлено:

- Метрики пересняты: 269 тестов (было 265; прогон подтверждён — зелёные за 5.2 сек), 213 полей конфига по AST (было «~222»), 360 модулей всего с учётом routers/prompts/ssr_ai/pages (было 329 без них), 48.7 МБ/209 скриншотов точно.
- Убрана привязка к снапшоту working tree (устаревает за часы), добавлена оговорка о сроке годности метрик.
- «CI/CD работает» смягчено до «workflow настроен, успешный run на GitHub требует подтверждения».
- П5 (зависимости): добавлены поправки — `ruff` отсутствует в extra `dev`; `FlagEmbedding` — core-зависимость, extras её из CI не уберут.
- Keep-reason-модули: «удалить» → «owner-аудит по одному»; `index_backup` исключён (защищён правилом в `conventions_architecture.md`).
- Voice: уточнено, что удаление — мини-проект (sidebar + feature registry + Telegram handler + deps + доки), а не один файл.
- SSR ML: «заморозить» подтверждено, удаление обусловлено compatibility-аудитом (gated enrichment архитектурно допустим).
- Скриншоты: перенос обусловлен предварительным изменением doc-policy (`docs/index.md`, `docs/architecture.md` называют их runtime-витриной).
- Формулировка про тесты: «большинство UI-smoke» → «явные дыры вокруг тяжёлых сервисов» (алгоритмических тестов в suite достаточно).
- Зафиксировано размещение в `doc/archive/` как сознательный выбор по прецеденту `arch_review_2026-06-25.md`.
