# Эволюционный подход к разработке

Актуализировано: 2026-07-12.

Этот документ фиксирует инженерный стиль развития runtime-репозитория
`hometutor`: маленькие проверяемые волны вместо больших рискованных рывков.
Он дополняет [conventions.md](conventions.md), но не заменяет жёсткие правила
по config, provider-layer, prompts, persistence, UI и тестам.

## Суть подхода

Эволюционная разработка — это способ строить продукт через серию небольших
изменений, где каждая итерация:

- сохраняет работоспособность системы;
- добавляет наблюдаемую ценность;
- уважает существующие данные, привычки пользователя и архитектурные инварианты;
- оставляет следующему шагу больше ясности, чем было до него.

В `hometutor` это особенно важно: retrieval, tutor, quiz, flashcards, spaced
repetition, learner state, Streamlit UI и local-first persistence связаны в
один учебный контур. Большой «идеальный» рефакторинг легко нарушает доверие
к продукту. Эволюционная волна, наоборот, улучшает один конкретный путь и
доказывает это targeted-проверкой.

## Что такое волна

Волна — маленькая завершённая итерация вокруг конкретного разрыва в поведении
системы.

Пример:

1. Разрыв: learner model обновляется внутри tutor flow, но пользователь не
   видит, что учебный след записан.
2. Волна A1: провести `learner_trace` через postprocessing → metadata →
   persistence → history → renderer.
3. Волна A2: проверить честность trace и показывать UI receipt только после
   подтверждённого canonical update.
4. Волна A3: добавить roundtrip-тест, чтобы путь не сломался незаметно.

Это не «сразу построить идеальную память обучения», а постепенно сделать
контур видимым, честным и защищённым.

## Принципы

### Малые изменения

Изменение должно быть достаточно маленьким, чтобы его можно было понять,
проверить и безопасно продолжить. Если задача требует большой миграции, она
разбивается на последовательные волны.

### Рабочее состояние после каждой волны

После каждой итерации проект должен оставаться запускаемым и тестируемым.
Фраза «почти работает, потом допилим» создаёт долг, а не эволюцию.

### Сохранение инвариантов

Эволюционная волна не имеет права ломать базовые правила проекта. Примеры:

- `mastery_vector` хранит только canonical concept id активного графа;
- config читается через `get_settings()` / `get_retrieval_settings()`;
- LLM/embeddings идут только через provider-layer;
- `sessions_completed` относится к завершённым quiz/session-path, а не к
  tutor/flashcard событиям;
- user-state persistence идёт через `app/user_state*.py`, без ad hoc SQLite
  из UI/роутеров.

### Наблюдаемость

Если система стала умнее, это должно быть видно: в UI, metadata, диагностике,
логе или тесте. Невидимая «магия» быстро теряет доверие.

### Честность интерфейса

UI не должен обещать больше, чем реально сделал backend.

Писать «единая очередь» нельзя, если код считает сумму двух очередей. Писать
«след записан» нельзя, если canonical learner update не произошёл. Эволюционный
подход не скрывает незавершённость, а называет её точно.

### Backward compatibility

Новые поля и контракты добавляются так, чтобы старые callers и старые данные
продолжали жить. Предпочтительны optional metadata, `.get()`, безопасные
defaults и graceful degradation.

### Измеримость

У каждой волны должен быть ответ на вопрос: как мы узнаем, что стало лучше?
Обычно это targeted-тест, contract-test, smoke-test, UI receipt, метрика или
ясная диагностическая строка.

## Практический чек-лист волны

Перед началом:

- Сформулировать один конкретный разрыв в поведении.
- Найти минимальный runtime write-set.
- Проверить, какие инварианты могут быть затронуты.
- Определить targeted-тесты до начала правок.

Во время реализации:

- Менять только файлы, необходимые для волны.
- Не делать попутный рефакторинг.
- Сохранять старые контракты и defaults.
- Делать UI/metadata формулировки честными.
- Не добавлять LLM, feature flag или новый слой, если достаточно
  deterministic/state изменения.

Перед завершением:

- Добавить или обновить targeted-тест на пользовательский/контрактный путь.
- Прогнать только релевантные тесты.
- Прогнать `ruff` по изменённым Python-файлам, если применимо.
- В финале явно назвать, что изменилось, что проверено и что осталось
  следующей волной.

## Антипаттерны

- Большой рефакторинг «заодно».
- Новая абстракция до того, как боль стала повторяемой.
- UI обещает больше, чем делает backend.
- Тест проверяет внутреннюю деталь, но не пользовательский контракт.
- `mastery_vector` загрязняется human labels, тегами или именами файлов.
- Документация говорит «единая очередь», а код считает сумму.
- «Работает на happy path», но не защищено от частичного отказа.
- Runtime-документ ссылается на локальный файл вне репозитория без явного
  указания, что это материал `hometutor-studio`.

## Хороший критерий завершения

Волна считается хорошей, если после неё можно честно сказать:

> Мы улучшили конкретный путь. Старые пути не сломали. Инварианты сохранили.
> Поведение стало честнее или видимее. Есть targeted-проверка. Следующий шаг
> стал понятнее.

Это спокойный инженерный стиль: меньше зрелищности, больше доверия к системе,
данным и будущим изменениям.
