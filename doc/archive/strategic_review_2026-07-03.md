# Стратегический обзор hometutor — что развивать, а что нет

**Дата:** 2026-07-03
**Тип:** глубокий анализ приложения, архитектуры, документации, тестов, зависимостей и гигиены репозитория.
**Предыдущий аудит:** [arch_review_2026-06-25.md](arch_review_2026-06-25.md) (часть находок закрыта регрессионными гвардами — учтено ниже).
**Состояние на момент обзора:** ветка `main` @ 957bcc2 + незакоммиченные правки в `app/ui/mission_control.py`, `app/ui/living_konspekt_view.py`, `app/ui/dashboards_graph.py`, `tests/`.

---

## TL;DR

Проект в необычно хорошей форме для своего размера: границы архитектуры чистые и **охраняются автоматическими гвардами**, документация системная и актуализируется по датам, тесты зелёные и быстрые (265 за 5 сек), CI/CD работает. Главные риски — не в качестве, а в **масштабе и расползании**: ~86 600 строк кода в ~330 модулях на локальное однопользовательское приложение, 222 поля конфигурации, 27 модулей только под Smart Study Router, и гвард размера **уже сломан текущими незакоммиченными правками**.

Стратегический вывод: **прекратить наращивать поверхность (новые фичи, ML, интеграции) и инвестировать в ядро учебного цикла, консолидацию и тесты сервисного слоя.**

## Что уже сильно — продолжать в том же духе

**1. Самоохраняющаяся архитектура.** Самое ценное в проекте. После аудита 25.06 появились регрессионные гварды (`scripts/arch_regression_guards.py`): контроль доступа к конфигу (`check_config_access.py`), детектор мёртвых модулей (`check_dead_modules.py`), проверка requirements против импортов (`check_requirements_imports.py`), бюджет размера (`check_size_budget.py`). Границы подтверждаются: все LLM-клиенты через `app/provider.py`, промпты централизованы в `app/prompts/`, backend не импортирует UI, никакого ad-hoc SQL в роутерах/UI, SQL-идентификаторы через allowlist. **Продолжать: каждое новое архитектурное правило — сразу с гвардом.**

**2. Документационная система.** `docs/index.md` как навигатор с ролями, таблица «источников истины», явная политика документации, датированные актуализации с журналом исправленного дрейфа. Уровень зрелых команд. **Продолжать актуализацию при каждом изменении маршрутов/поведения.**

**3. Ядро продукта — учебный цикл.** Ответ с источниками → tutor → quiz → flashcards → SM-2 → mastery → адаптивный план → SSR. Цельная, отличимая от «чата с PDF» идея. **Всё развитие должно углублять этот цикл, а не добавлять параллельные поверхности.**

**4. Работа над видимостью UI (в процессе).** `app/ui/feature_registry.py` с 4 уровнями опыта, `docs/first_run_experience_plan.md`, Mission Control с preflight-проверками. Правильный ответ на главную продуктовую проблему — перегруз интерфейса ~19 фичами. **Довести до конца в первую очередь.**

**5. Аутентификация и безопасность.** Опциональный JWT + bcrypt с prod-гвардом против dev-секрета (`guard_insecure_jwt_secret`), изоляция state через `contextvars` + резолв пути без переписывания схемы — элегантно. Log-masking подключён и покрыт тестом (`tests/test_logging_invariants.py`). Confirmation-токены на деструктивные скрипты. Local-first с loopback-дефолтами. **Достаточно — не наращивать (роли, OAuth и т.п. этому продукту не нужны).**

**6. CI/CD.** ruff + pytest на каждый push, автодеплой на HF Spaces с мягким скипом без секретов. Работает — не трогать, кроме пункта П5 ниже.

## Что развивать в первую очередь (приоритетный список)

**П1. Починить сломанный гвард размера — и не «поднятием бюджета».** Прямо сейчас `check_size_budget.py` падает: незакоммиченные правки в `app/ui/mission_control.py` и рост `app/ui/knowledge_graph_d3.py` (1720 строк) превысили бюджеты. Судя по значениям, бюджеты выставлены «по пиковому нарушителю» — храповик крутится не в ту сторону. Нужно: при каждом касании файла-нарушителя выносить куски и **снижать** бюджет после каждого улучшения.

**П2. Тесты сервисного слоя.** 3 500 строк тестов на 86 600 строк кода, большинство — UI-smoke. Инвариантные тесты pipeline/guardrails/logging — отличное начало, но целевых тестов заслуживают:
- `app/query_service.py` (960 строк);
- `app/tutor_orchestrator.py` (696);
- `app/flashcard_service.py` (994, включая SM-2 edge cases);
- `app/index_lifecycle.py` / `app/index_registry.py` — целостность индекса — единственные данные, которые больно терять.

**П3. Консолидация вместо роста.** Конкретные цели:
- **SSR: 27 модулей** (`ssr_*`, `smart_study_*`, `ssr_ai/`) — при задекларированном «deterministic-first». Слить экспериментальные/gated-слои, оставить: сигналы → скоринг → объяснение.
- **Eval: 8 модулей** (`eval_baseline`, `eval_uplift`, `eval_ragas_backend`, `eval_retrieval_comparison`, `router_eval`, `compare_eval`…) — один harness с бэкендами вместо шести параллельных.
- **«Мёртвые с keep-reason» модули**: `check_dead_modules.py` превратился в белый список кладбища — 16 модулей числятся «на будущее» (`ssr_pregeneration`, `answer_parser`, `tutor_context_parser`, «kept for API evolution»…). Удалить: git хранит историю, а каждый лишний модуль дорожает при каждом ревью и рефакторинге.

**П4. Сжать конфигурационную поверхность.** 222 поля настроек и 74 env-ключа — поверхность отказов и документационного дрейфа. Пройтись по флагам, которые всегда в дефолте, и убрать вместе с ветками кода. Эвристика: если флаг не переключался ни разу с момента создания — это не конфигурация, а мёртвый код с усложнённым интерфейсом.

**П5. Единый источник зависимостей.** Сейчас два: `requirements.txt` (всё вперемешку, включая optional) и `pyproject.toml` (аккуратные extras: voice/ocr/otel/telegram/ml). CI ставит полный `requirements.txt` — включая docling, pymupdf, FlagEmbedding — на каждый прогон. Перевести CI и README на `pip install -e .[dev]`; `requirements.txt` либо удалить, либо генерировать.

## Что точно НЕ нужно развивать (stop-list)

**1. ML-надстройки над SSR.** `ssr_ml_reranking`, `ssr_ml_monitoring`, `scripts/ml/` с обучением forgetting curve — ML-инфраструктура промышленного масштаба для **одного пользователя с крошечным датасетом**. Кривая забывания одного человека не даст данных для обучения ничего лучше SM-2, который уже есть. Заморозить; детерминированный SSR — правильная и достаточная архитектура.

**2. Voice-фича.** `app/voice_service.py` + SpeechRecognition + pyttsx3 в зависимостях ради одной точки в сайдбаре. Для учебного RAG-приложения на Streamlit ценность маргинальная, зависимости тяжёлые и платформозависимые. Кандидат на удаление целиком.

**3. Telegram-бот как вторая продуктовая поверхность.** Каждая фича должна работать в двух UI. Если бот не используется ежедневно — заморозить на уровне «уведомления о повторениях», учебный цикл туда не переносить.

**4. Enterprise-обсервабилити.** 8 модулей `metrics_*`, SLO, Langfuse-экспорт, OTel-трейсинг, `latency_budget`, `pipeline_profiler` — оборудование самолёта на велосипеде. Не удалять то, что подключено и работает, но **не расширять**: ни новых SLO-дашбордов, ни новых экспортёров.

**5. Новые режимы retrieval и новые ручки.** Документация сама формулирует правило: «Do not add public retrieval modes when a bounded RAG profile is enough». Соблюдать. Hybrid (Chroma+BM25) + reranker + graph expansion уже выше среднего по качеству.

**6. Новые views в Streamlit UI.** 107 UI-модулей, 19 фич в реестре, и главная текущая работа — *прятать* сложность за уровнями опыта. Добавление 20-й view работает против этой цели. UI-бюджет тратить на консолидацию и полировку tier 1–2.

**7. Расширение auth.** Опциональный JWT сделан для рубрики проектной работы и сделан хорошо. Роли, OAuth, refresh-токены, admin-панели — не нужны local-first продукту.

## Гигиена (исправить один раз и забыть)

1. **50 МБ скриншотов в git** (`docs/screenshots/`, 209 файлов) — весь вес репозитория. Перенести в `hometutor-studio` (там уже живут сценарные манифесты) или в Git LFS/release assets; в `docs/quickstart_demo.md` оставить ссылки. Клонирование и CI-checkout станут в разы легче.
2. **Сообщения коммитов-числа** («56», «55»). История не помогает ни bisect'у, ни ревью. Писать `тип(область): что и зачем` — примеры в репо уже есть (`docs: sync architecture...`).
3. **Глобальные ruff-игноры маскируют реальные проблемы.** `F821` (undefined name!) и `F401/F403/F405` выключены на весь репозиторий из-за star-реэкспортов в двух файлах (`app/api_models.py`, `app/user_state.py`). Перенести в `per-file-ignores` — сейчас линтер слеп к опечаткам в именах во всех 330 модулях.
4. **Процессные документы в runtime-репо вопреки собственной политике.** `docs/index.md` декларирует: процессные материалы — в `hometutor-studio`. При этом в `docs/` лежат `compliance_upgrade_plan.md`, `first_run_experience_plan.md`, `ui_visibility_phase1_plan.md` — handoff-задания. По завершении работ — в архив или в studio.
5. Мелочи: `.codex_tmp/` в рабочем дереве (не добавлен в `.gitignore`); кириллические имена файлов страниц (`3_Мой_прогресс.py`) дают quoted-пути в git на Windows; `models/` — пустая папка с `.gitkeep`.

## Рекомендуемый порядок действий

1. **Сейчас:** довести до конца работу по видимости UI / first-run (в процессе, незакоммичена) и починить упавший size-гвард в этих же файлах.
2. **Ближайший спринт:** удалить мёртвый кластер (voice, keep-reason-модули), слить eval-harness'ы, перевести CI на `pyproject` extras, вынести скриншоты из git.
3. **Фоново, по правилу «тронул — улучшил»:** дробление файлов >600 строк со снижением бюджета, тесты на `query_service` / `tutor_orchestrator` / `flashcard_service` / `index_lifecycle`.
4. **Постоянно:** любое предложение новой фичи проверять вопросом «углубляет ли это цикл ответ→tutor→quiz→cards→progress→SSR?» Если нет — в studio-backlog, не в код.

---

**Суть в одном предложении:** у проекта отличный скелет и работающая иммунная система (гварды, доки, CI) — угроза не в качестве, а в том, что тело растёт быстрее, чем скелет успевает его держать. Побеждать надо не добавлением, а вычитанием.
