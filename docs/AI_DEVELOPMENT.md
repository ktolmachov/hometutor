# AI Development — как ИИ использовался при разработке hometutor

Этот документ показывает использование ИИ-ассистентов (Cursor agent на базе LLM) на каждом
этапе работы над `hometutor`: от планирования до отладки. Часть истории — планирование,
backlog, user stories, сценарные манифесты — ведётся в соседнем репозитории
**[`hometutor-studio`](https://github.com/ktolmachov/hometutor-studio)**; здесь приводятся
кросс-ссылки и конкретные примеры, относящиеся к runtime-коду этого репозитория.

## 1. Планирование

Идея продукта, анализ конкурентов, user stories и backlog ведутся в
`hometutor-studio` (YAML-манифесты сценариев, `doc/backlog_registry.yaml`-эквивалент,
демо-генератор). В `hometutor` ИИ использовался для перевода согласованных в чате
требований в исполняемый план:

- Документ [docs/compliance_upgrade_plan.md](compliance_upgrade_plan.md) — пример того, как
  устное согласование («миграция на PostgreSQL не нужна», «FastAPI + JWT + bcrypt»,
  «CI/CD: линт + pytest + автодеплой», «Яндекс.Метрика») было превращено ИИ в детальный
  план с точными путями файлов, сигнатурами функций, эскизами кода, тестами и
  acceptance-критериями — план был написан так, чтобы его можно было исполнить другой
  моделью «вслепую», без дополнительных уточнений у пользователя.
- Перед написанием плана ИИ-агент самостоятельно изучил код (`app/user_state_db.py`,
  `app/api.py`, `app/api_auth.py`, `app/ui_client.py`, `app/ui/main.py`), чтобы план
  опирался на реальные точки расширения, а не на предположения.

## 2. Дизайн/архитектура

Ключевое архитектурное решение по аутентификации было выработано ИИ через анализ
существующего кода, а не «с нуля»:

- **Проблема:** нужна изоляция learner state по пользователю, но в репозитории ~24 таблиц
  state (`reading_status`, `flashcards`, `spaced_repetition`, `learner_profile_snapshots`, …),
  переписывать каждую под `user_id`-колонку — overengineering для объёма задачи.
- **Решение ИИ:** вся state-БД резолвится через одну точку — `app/user_state_db.py::_connect()`.
  Достаточно сделать путь к SQLite-файлу зависимым от текущего пользователя
  (`data/users/<uid>/user_state.db` вместо `data/user_state.db`) через `contextvars`,
  и изоляция работает для всех таблиц одновременно, без миграции схемы.
- Для реестра пользователей спроектирована отдельная глобальная БД (`auth.db`) с тремя
  связанными таблицами: `users` ← `auth_sessions`, `auth_audit_log` (FK `ON DELETE CASCADE`/
  `SET NULL`) — закрывает требование «3+ связанные таблицы» без раздувания схемы state.
- Для HF Spaces ИИ сравнил Streamlit SDK и Docker SDK Hugging Face Spaces и выбрал Docker SDK,
  аргументировав это тем, что Streamlit-SDK Space не запускает параллельно FastAPI, а UI
  ходит в API по HTTP — без Docker REST API был бы недоступен на демо.

## 3. Разработка — примеры «промпт → результат»

| № | Задача (промпт пользователю/себе) | Результат |
|---|---|---|
| 1 | «Добавь per-user изоляцию state без переписывания 24 таблиц» | Рефактор `_connect()` → `_resolve_state_db_path()` в `app/user_state_db.py`, новый `app/auth_context.py` (contextvar), без изменений в таблицах state. |
| 2 | «JWT + bcrypt, регистрация/логин, привязка к user_id, не ломая текущие тесты» | `app/auth_service.py`, `app/auth_db.py`, `app/auth_models.py`, `app/routers/auth.py` + флаг `AUTH_ENABLED=false` по умолчанию — старое поведение и все существующие тесты остаются зелёными. |
| 3 | «Streamlit должен требовать логин, если включена аутентификация» | `app/ui/auth_gate.py` (формы вход/регистрация, `require_ui_auth_or_stop()`) + подключение в `app/ui/main.py` и во всех страницах `app/ui/pages/*.py`. |
| 4 | «Добавь тесты на регистрацию/логин/401/409/422 и на изоляцию state между пользователями» | `tests/test_auth.py` — 11 тестов (happy path, неверный пароль → 401, дубликат email → 409, слабый пароль/email → 422, просроченный/битый JWT → 401, изоляция state между двумя пользователями, регресс при `AUTH_ENABLED=false`). |
| 5 | «Подготовь CI: линт + тесты, без падения на существующем коде» | `.github/workflows/ci.yml` + стартовая конфигурация `[tool.ruff]` в `pyproject.toml` — выявлено 575 существующих предупреждений (`F401/F403/F405` от намеренных `import *`, `E402` от намеренного `sys.path.insert`, `E501` от длинных RU-комментариев); ИИ принял решение не рефакторить весь репозиторий ради линтера, а явно задокументировать и проигнорировать эти категории, оставив проверку значимой для нового кода. |
| 6 | «Инъекция Яндекс.Метрики без поломки локалки» | `app/ui/analytics.py` — идемпотентный патч `index.html` Streamlit по маркеру-комментарию, no-op без `YANDEX_METRIKA_ID`, с тестами (`tests/test_yandex_metrika.py`) на фейковом `streamlit`-модуле, без касания реального пакета. |

## 4. Backend/инфраструктура

- **Схема БД:** ИИ спроектировал и применил схему `auth.db` (3 таблицы + 2 индекса) по
  образцу уже существующего паттерна подключения (`_with_db` helper, `WAL`, `_DB_WRITE_LOCK`)
  из `app/user_state_core.py`, вместо изобретения нового стиля доступа к SQLite.
- **Эндпойнты:** `POST /auth/register`, `POST /auth/login`, `GET /auth/me`, `POST /auth/logout`
  спроектированы как отдельный публичный роутер (`app/routers/auth.py`), подключённый в
  `app/api.py` рядом с `core_router`; защищённые роутеры получили вторую зависимость
  (`Depends(auth_scope)`) рядом с существующей `Depends(require_api_key)`, без замены.
- **CI/CD:** `.github/workflows/ci.yml` (ruff + pytest на push/PR в `main`) и
  `.github/workflows/deploy.yml` (push в Hugging Face Space через `workflow_run` после
  успешного CI, с защитным skip при отсутствии секретов `HF_TOKEN`/`HF_USERNAME`).
- **Docker/деплой:** `deploy/docker/docker_entrypoint.sh` дополнен вызовом
  `bootstrap_demo_paths.sh` (подготовка demo-данных на эфемерном FS), корневой `README.md`
  получил YAML-заголовок для HF Docker SDK (`sdk: docker`, `app_port: 8501`).

## 5. Отладка/оптимизация

- **Скрытая проблема с потоками:** Tutor-чат запускает `query_service.answer_question` в
  `ThreadPoolExecutor` (`app/ui/tutor_chat_session.py`) для отзывчивости UI. ИИ заранее
  определил риск: `contextvars` (включая `user_id` из `app/auth_context.py`) не наследуются
  новым OS-потоком — без явного проброса worker-поток писал бы в общий `user_state.db`,
  а не в per-user файл, что незаметно ломало бы изоляцию именно в самом нагруженном
  пользовательском сценарии. Исправлено через `contextvars.copy_context().run(...)`
  при `executor.submit(...)`.
- **Lint-шум на легаси-коде:** прогон `ruff check app tests` без конфигурации показал 575
  предупреждений. ИИ итеративно сузил `[tool.ruff.lint]` до значимого набора (`E`, `F`)
  с явно задокументированными игнорами под существующие паттерны репозитория, проверяя
  после каждого изменения результат (`ruff check --statistics`), вместо одной большой
  непрозрачной правки.
- **Регрессии:** после каждого функционального блока запускался прицельный набор тестов
  (`pytest tests/test_auth.py`, затем полный `pytest tests/` — 61/61 зелёных, включая
  50 ранее существующих тестов без единого изменения в их коде).

## 6. Проблемы и решения

1. **Проблема:** изоляция state по пользователю рискует требовать миграции 24 таблиц.
   **Как помог ИИ:** нашёл единственную точку резолва пути к БД и сделал её per-user —
   изоляция «бесплатно» для всех таблиц. **Итог:** ноль изменений в схеме state-таблиц.
2. **Проблема:** новая аутентификация могла сломать 50 существующих тестов, не использующих
   токены. **Как помог ИИ:** ввёл флаг `AUTH_ENABLED` (default `false`) на каждом уровне
   (API-зависимость, UI-гейт). **Итог:** 61/61 тестов зелёные (50 старых + 11 новых).
3. **Проблема:** worker-поток тьютора рисковал терять контекст пользователя незаметно
   (без падения теста — баг проявился бы только в проде с двумя одновременными
   пользователями). **Как помог ИИ:** заранее (до релиза, не после инцидента) нашёл и
   пробросил контекст через `copy_context()`.
4. **Проблема:** включение линтера на легаси-репозитории грозило либо нерелевантным шумом
   (575 ошибок), либо широким нежелательным рефакторингом. **Как помог ИИ:** предложил
   прагматичный baseline-конфиг с явной документацией причин каждого игнора.
5. **Проблема:** старый `deploy/hf-spaces/README.md` описывал Streamlit SDK, который не
   поддерживает FastAPI — это противоречило плану (UI зависит от API). **Как помог ИИ:**
   сравнил два SDK Hugging Face Spaces и зафиксировал решение (Docker SDK) с обоснованием
   в документации, обновив инструкцию и секреты.
6. **Проблема:** реальный деплой на Hugging Face и настройка GitHub Secrets требуют внешних
   учётных данных, недоступных агенту в изолированной среде. **Как помог ИИ:** честно
   зафиксировал это ограничение, подготовив весь необходимый код (workflows, Dockerfile,
   YAML-заголовок, инструкции) так, чтобы пользователь мог завершить деплой самостоятельно
   за несколько шагов, без дополнительной разработки.
7. **Проблема:** Яндекс.Метрику невозможно штатно вставить в `<head>` Streamlit-страницы.
   **Как помог ИИ:** нашёл обходной путь (патч раздаваемого `index.html` по маркеру) и
   закрыл его тестами на изолированном fake-модуле, не трогая реальную установку Streamlit.

## 7. Выводы и рекомендации

- Использование ИИ-агента наиболее эффективно там, где сначала проводится **разведка кода**
  (grep/read конкретных точек расширения), а затем — **точечное изменение** с сохранением
  обратной совместимости через feature-флаги (`AUTH_ENABLED`, `YANDEX_METRIKA_ID`).
- Для легаси-репозиториев внедрение линтера/CI эффективнее как **baseline с явными
  игнорами и комментариями**, чем как блокирующий полный рефакторинг — иначе высок риск
  непропорционально большого изменения вне исходной задачи.
- Скрытые баги конкурентности (contextvars в worker-потоках) — то, что ИИ-агент должен
  искать **проактивно** при добавлении любого per-request/per-user состояния в код с
  фоновыми потоками, а не дожидаться обнаружения вручную.
- Часть задачи (реальный внешний деплой, секреты) не может быть выполнена ИИ-агентом в
  изолированной среде — рекомендация: агент должен явно фиксировать такие границы в плане
  и в документации, а не имитировать выполнение.
