# Живой конспект: анализ упущений и план следующих волн

Статус: аналитический handoff-документ (самодостаточный). Ревизия: 2026-07-06, четвёртый
проход — критическая сверка реализации W4 с планом и детализация следующего хода:
W5 «Живой артефакт» (см. подраздел в §7 и журнал в конце).
Подготовлено: 2026-07-06, после волны «вкладки + вход изнутри + reader + staleness +
медиа-артефакт» (3 раунда аудита; верификация — в PR/commit message соответствующих
коммитов, а не точным счётчиком тестов в этом документе — счётчик устаревает за день).
Родственные: `docs/multimodal_konspekt_plan.md`, `docs/adr/0001-multimodal-media-contract.md`,
`docs/living_konspekt_masterpiece_council.md` (критерии шедевра, стол пользователей,
три продуктовые ставки поверх волн — «фрагменты», синхро-чтение, «Спроси лектора»).

---

## 1. Где фича сейчас

Работает: корзина разделов с автоперсистом (`app_kv`) через `workbench_service`,
row-контракт v2 (`row_key`, переносимые `*_rel` пути, non-portable snapshot), вход изнутри
(обзор + поиск), 5 вкладок (Разделы → Читать → Память → Сохранить → Дальше), порядок глав,
staleness-снимков, медиа-таймкоды в UI и в сохранённом артефакте (с правилом доверия),
term-cards, память по `source:`-тегам, deep-study промпт. Входы: Граф, Flashcards, панель
добавления, resume-карточка Mission Control.

Этот документ — о том, чего **нет**, и что из этого важно.

### Коротко простыми словами

Сейчас «Живой конспект» уже ездит: пользователь может набрать полезные куски
лекций, прочитать их, сохранить и вернуться к материалу. Но пока это скорее
корзина сильных фрагментов, чем полноценный учебный инструмент.

Цель следующих волн: превратить сценарий в понятный цикл обучения —
**собрал → прочитал → переработал своими словами → проверил себя → сохранил →
вернулся позже**. Поэтому порядок важен: сначала укрепляем фундамент
(`workbench_service`, переносимые пути, `row_key`, миграции), потом делаем
сохранённый конспект живым артефактом с manifest/round-trip, и только после этого
добавляем пользовательские заметки, прогресс чтения и quiz по своей сборке.

## 2. Критические упущения — архитектура

### A1. Бизнес-логика корзины живёт в UI-слое (главный долг)

`add/move/remove/persist/hydrate` определены в `app/ui/living_konspekt_view.py`. Точная
формулировка проблемы: **domain-helpers смешаны со Streamlit-view и импортируются внешними
UI-модулями** — `dashboards_graph`, `flashcards_review_view`, `mission_control`, `sidebar`,
`main` (при бутстрапе UI вызывает `ensure_workbench_hydrated` — это доменная гидрация/CRUD,
не рендер-цикл фичи). Импорты из `add_panel`/`reader`/`fragments` — внутренние для фичи и
рендер-циклы, их в счёт долга не записываем. Это нарушает правила проекта: «Keep UI thin: business logic
belongs in services» (`docs/architecture.md`, §Architecture rules) и «доменные сервисы должны
быть переиспользуемыми из FastAPI, Streamlit и Telegram» (`docs/conventions.md`). Следствия:
корзину нельзя открыть через API/Telegram/CLI, тестовый DI-параметр `state` — костыль вокруг
`st.session_state`, каждый новый потребитель усиливает связку с Streamlit.

**Решение:** `app/workbench_service.py` — перенос CRUD + персиста (хранение уже в `app_kv`,
т.е. сервис-слой готов принять). Реэкспорты в view — только как явно временный шаг миграции,
не постоянное решение. **AC волны:** `dashboards_graph`, `flashcards_review_view`, `mission_control`, `sidebar`,
`main` импортируют сервис напрямую (не через `living_konspekt_view`);
реэкспорты либо удаляются в той же волне, либо помечаются `# TODO(W4-cleanup)` с тестом,
который явно проверяет их пустоту/удаление — иначе долг «домен в UI-файле» не закрыт, а
просто переименован.

**Пост-W4 уточнение (четвёртый проход):** в view остались не реэкспорты домена, а тонкие
session_state-адаптеры (`add/move/remove_section_*` — целевая модель «stateless-сервис +
зеркало» из ADR 0003, §Live-state model); требование «тест на пустоту» к ним неприменимо.
Остаточный долг сведён к переводу внутренних импортов фичи (`living_konspekt_add_panel`)
на общий адаптерный слой и снятию `TODO(W4-cleanup)` формулировкой, а не удалением
функционального кода.

### A2. Абсолютные пути ломают переносимость (тихая потеря данных)

Проверено: `app_kv` **входит** в sync-whitelist (`user_state_db._SYNC_TABLE_COLUMNS`) — корзина
едет в backup/QR-переезд. Но строки корзины хранят `konspekt_md_abs` — **абсолютные пути**
(`D:\AI\app\data\...`). После переезда на другую машину/смены DATA_DIR каждый deep-link,
staleness, медиа-панель и память конспекта молча ломаются, хотя данные «синхронизировались».

**Решение:** версионировать **весь** row-контракт, а не одно поле, и явно развести две модели —
иначе миграция чинит sync и ломает UI, который сегодня напрямую использует abs-пути для dedup,
deep-links, staleness и Obsidian/VS Code (`section_index.py`, `living_konspekt_view.py`):

- **Persisted row (хранится в `app_kv`/research payloads):** `row_key` +
  `konspekt_md_rel` + `source_rel` (DATA_DIR-relative, как в media-sidecar) +
  `row_version`. Ничего абсолютного на диске. Полная portable/non-portable схема
  (с `portability_status`, `*_label`, `resolve_error`) и контракт резолва — в
  **ADR 0003**; этот план фиксирует только инварианты.
- **Runtime row (то, что видит UI после гидрации):** сервис резолвит rel → abs **на лету** и
  отдаёт workbench runtime row: сегодняшние abs-поля (`konspekt_md_abs`/`source_abs`)
  остаются вычисленными для deep-links/staleness/media, а операции идентичности
  (add/dedup/move/remove) переходят на `row_key`. Путь вне текущего DATA_DIR при
  резолве → строка помечается непереносимым снимком (staleness-подсказка), а не падение.
- Fallback для строк, сохранённых до миграции (уже содержат abs) — конвертируются в rel лениво
  при первой гидрации, если путь внутри текущего DATA_DIR.

Та же миграция и то же разделение persisted/runtime — для payload'ов research-сессий (они
сериализуют те же rows). Контракт (обе модели, `row_version`, точка резолва) фиксируется в
ADR 0003 до реализации — сервис должен быть единственным местом, которое видит persisted-форму.

### A3. Артефакт умирает в момент сохранения

`_save_living_konspekt` пишет `# title + body` — **без frontmatter**. Текущий артефакт уже
человекочитаем в этом смысле: `_stitch_verbatim` вставляет per-section таймкод-строки
`*🎬 …*` (`_media_line_for_row` в `source_block` каждой строки) и один общий блок
«🎬 Видео материалов» (`_videos_block`) в конце артефакта после всех разделов — см.
`user_guide.md`. Потеря не в исчезновении таймкодов из текста.

Реальная потеря — в отсутствии **machine-readable** слоя: файл не имеет
`type: living-konspekt` (неотличим от обычного md), нет структурированного провенанса строк
(только текстовые «файл:строки», не парсимые обратно), нет интерактивного media-resolve (открой
файл через панель добавления как обычный документ — таймкоды остаются текстом, не кликабельным
UI-виджетом с проверкой staleness), и нет round-trip обратно в корзину. Плюс повторное сохранение
плодит `konspekt-1.md`, `konspekt-2.md` вместо обновления. Фича называется «живой», но у
артефакта нет пути назад в приложение — только вперёд, в vault.

**Решение (волна «Жизненный цикл»):** frontmatter + **`artifact_manifest_v1`** — новый контракт,
сознательно НЕ переиспользующий имя/схему `media_sidecar`: sidecar привязан к одному конспекту и
одному первичному медиа (`media_sha256`, таймкоды его секций), а сборный артефакт агрегирует
несколько документов и видео — это другой инвариант. Манифест несёт: `type: living-konspekt`,
компактный провенанс строк (для пересборки в корзину) и ссылки на sidecar'ы исходных конспектов
(таймкоды в читалке резолвятся через них, не копируются). Плюс: «Мои конспекты» (список
сохранённых с открытием/пересборкой), режим «обновить существующий» вместо копий `-1`/`-2`,
CTA переиндексации после сохранения (сейчас — только надпись, без кнопки).

### A4. Один глобальный workbench

Реальный студент ведёт 2–3 темы параллельно. Сейчас — одна корзина; именованные research-сессии
в сайдбаре — снимки с ручным save/restore и слабой находимостью. **Решение:** мульти-проекты
first-class (селектор активного конспект-проекта в шапке фичи: имя + цель). Research sessions —
общий механизм снимков в сайдбаре, не специфичный для Живого конспекта; депрекируется не
механизм целиком, а **использование его Живым конспектом** — после миграции на проекты
Living-Konspekt-снимки в этом UI скрываются/переносятся, сам механизм остаётся доступен другим
потребителям.

## 3. Упущения в методологии работы пользователя (process)

Это самый недооценённый пласт: фича собирает **чужой** текст, а конспектирование по данным
науки об обучении работает через **переработку своими словами** (generation effect, elaboration).

| Упущение | Метод, который это закрывает | Что добавить |
|---|---|---|
| Нет поля «моя мысль» у раздела | Cornell notes (колонка своих формулировок), Zettelkasten | Заметка пользователя на раздел (хранится в строке корзины; в артефакте — блок «💬 Моими словами» под разделом) |
| Нет цели конспекта | Goal-directed learning | Поле «цель» у конспект-проекта; едет в deep-study промпт и в заголовок артефакта |
| Вкладки — поверхности, не процесс | Учебный цикл | Мягкая воронка-подсказка: Собрал → Прочитал → Переработал → Закрепил → Повторяю (прогресс-стрип в шапке; SSR-сигнал «конспект собран, но не закреплён») |
| Quiz по корзине отсутствует | Retrieval practice по СВОЕМУ материалу | `quiz_scoped` существует, но «просто подключить» нельзя: нужен адаптер scope-контракта (workbench rows → стабильные text/source refs → генерация). **AC:** вопросы строятся строго из текстов собранных строк, а не из повторно найденных секций индекса |
| Reader пассивен | Активное чтение | Позиция чтения (персист), «отметить прочитанным» по разделам, прогресс «прочитано 7/20» |
| «Проверь себя» — только дословные вопросы лектора | Interleaving/elaborative interrogation | Кнопка «сгенерировать 3 вопроса по моей сборке» (LLM, opt-in) |

Ключевой принцип волны: **не добавлять новые поверхности — добавлять этапы переработки в
существующие вкладки.**

## 4. Упущения UI

1. **«Мои конспекты»** — сохранённые артефакты не перечислены нигде; попасть в них можно только
   через vault/поиск. Нужен список во вкладке «Сохранить» (имя, дата, разделов, открыть/пересобрать).
2. **Печать/PDF** — `app/ui/print_view.py` подключён глобально (`fragments.py`, `main.py`,
   topic/plan-потоки), но **не как цель экспорта Живого конспекта**; собранный конспект — главный
   кандидат на «чистый вид» (подготовка к экзамену без экрана).
3. **TOC в reader** — 20+ разделов без оглавления/якорей; нужен сайд-список заголовков.
4. **Bulk-операции корзины** — «очистить всё», «убрать разделы документа X»; сейчас только по одному.
5. **Вход из «Быстрого ответа»** — перенос из прошлого плана: «➕ В конспект» на карточке
   источника. НЕ рецепт 1:1 с flashcards: карточка там уже знает свой раздел, а source card —
   это retrieval-чанк без надёжного якоря. **AC:** резолв через `top_sections_for` с порогом
   уверенности; ниже порога или при дублях заголовков — не тихое добавление, а выбор из
   кандидатов («какой раздел вы имели в виду?»).
6. **Onboarding** — фича tier 3 без главы в `tutorial_chapters.py`; для killer-фичи нужна глава
   гида + demo-сценарий в витрине (`docs/quickstart_demo.md`).
7. **Drag-and-drop порядка** — `↑/↓` не масштабируется на 30 разделов; `st.data_editor` с
   порядковой колонкой или нумерация «переместить на позицию N».

### W5 «Живой артефакт» — детальный состав (следующий ход)

Цель волны словами пользователя: **«я сохранил конспект — и он не умер»**. Сохранённый
артефакт виден в приложении, обновляется вместо копий и возвращается в корзину одним кликом.

Ядро (контракт → потом UI):

1. **`app/konspekt_artifact.py`** — доменный владелец формы артефакта (правило то же, что для
   `workbench_service`: не импортирует `streamlit`/`auth_gate`). Сюда переезжают
   `_stitch_verbatim`, `_save_living_konspekt` и доменная часть media-строк артефакта
   (чистые helpers; UI-панель остаётся в `living_konspekt_media`) — это и закрывает
   перенесённый size-budget view ≤600.
2. **`artifact_manifest_v1`** (frontmatter): `type: living-konspekt`, `artifact_id`
   (стабильный, генерируется при первом сохранении), `title`, `created_at`/`updated_at`,
   `row_version` и провенанс строк: `row_key`, `konspekt_md_rel`, `line_start`/`line_end`,
   `section_id` (`media_alignment.compute_section_id` — контентный якорь для пересборки при
   дрейфе строк), `note`/`read_at` как opaque passthrough; ссылки на media-sidecar'ы исходных
   конспектов — rel-путями. **Правило A2 распространяется на манифест: никаких abs-путей.**
3. **Save-as-upsert:** повторное «Собрать и сохранить» того же `artifact_id` обновляет файл
   (bump `updated_at`), а не плодит `-1`/`-2`; «Сохранить как новый» — отдельное явное действие.
4. **«Мои конспекты»** (вкладка «Сохранить»): скан `vault_root()/living-konspekt/*.md` с
   парсом манифеста — имя, обновлён, N разделов; действия «Открыть» (Obsidian/VS Code),
   «Обновить из корзины», «Пересобрать в корзину». Пересборка: manifest-провенанс →
   persisted rows → `normalize_runtime_rows`; строка, не найденная по `line_start`,
   ре-анкорится по `section_id`, иначе честно становится non-portable snapshot.
5. **Reindex-CTA:** после сохранения — кнопка «Обновить индекс» по готовому паттерну
   `POST /reindex` + `poll_reindex_status` (как в `query_tab_ask_panel`), а не только надпись.
6. **Print-цель:** «Чистый вид / печать» собранного артефакта через `print_view`.

Delight-пакет той же волны (мелкие, но пользовательски громкие):

7. **TOC в reader** (§4.3) — якорный список заголовков при 8+ разделах.
8. **Bulk-операции корзины** (§4.4) — «очистить всё», «убрать разделы документа X»;
   сервисные операции `remove_sections(row_keys)`/`clear_rows()` в `workbench_service`.
9. **Fix воронки:** `living_konspekt_section_added` сейчас трекается и при дубле-noop
   (`add_section_to_workbench` игнорирует `added` при трекинге) — событие должно уходить
   только при реальном добавлении, иначе метрики воронки W8 лгут.

AC волны:

- **Round-trip:** сохранить → «Пересобрать в корзину» → те же `row_key`/содержимое;
  `note`/`read_at` проносятся opaque (даже пустые/None).
- **Переносимость:** манифест без abs-путей; артефакт, сохранённый до W5 (без манифеста),
  остаётся валидным md — «Мои конспекты» показывает его как «без манифеста» (открыть можно,
  пересобрать нельзя), без миграции старых файлов.
- **Идемпотентность:** повторный save того же артефакта не создаёт новых файлов.
- **Size-budget:** `living_konspekt_view.py` ≤600 строк по итогам волны.

## 8. Анти-цели (чтобы волны не превратились в расползание)

1. **Не строить редактор markdown** — переработка это заметки-к-разделам, а не WYSIWYG; полный
   текст правится в Obsidian/VS Code (deep-links уже есть).
2. **Не делать realtime-коллаборацию** — local-first, один учащийся.
3. **Не добавлять новые вкладки** — этапы методологии встраиваются в существующие пять.
4. **Не плодить LLM-вызовы по умолчанию** — генерация вопросов/синтез остаются opt-in кнопками;
   ядро фичи детерминированно.
5. **Не расширять контракт строки без ADR** — после 0003 каждое поле проходит через ревизию ADR.
6. **Size-budget**: каждая волна обязана выносить больше, чем добавляет. Факт после W4:
   view — 859 строк (CRUD ушёл в сервис, но артефакт-сборка/склейка остались) — обещание
   «≤600 после W4» не выполнено; бюджет перенесён в AC W5 вместе с выносом артефакт-кода
   в `app/konspekt_artifact.py`.
